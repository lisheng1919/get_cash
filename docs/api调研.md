你的调研把miniQMT当工具使用没问题，但有一个关键设计需要理解清楚：miniQMT是标准版QMT的"极简模式"，**同一个QMT客户端在登录时勾选"极简模式"即可切换到miniQMT**。

两者对比：

| 维度 | 大QMT | 小QMT（你的选择） |
| :--- | :--- | :--- |
| **编辑器** | 内置Python编辑器 | 脱离客户端，VSCode/Jupyter任意写 |
| **第三方库** | 仅白名单库 | pip任意装（pandas/torch等） |
| **运行环境** | 客户端内（仅Windows） | 本地Python+客户端（Win/Linux/Mac） |
| **回测引擎** | 内置Tick→年线回测 | 无内置，需用backtrader等外部框架 |
| **适用场景** | 策略研究+回测+实盘一体化 | 策略已成熟，专注"稳定报单+外部灵活开发" |

- ✅ 你选miniQMT完全正确。LOF套利策略本身是成熟的，不需要内置回测引擎，你之前用backtrader离线回测更灵活，miniQMT专注做交易通道即可。

### 2. 云服务器部署与稳定性：社区用户的真实痛点

你未来如果要无人值守运行，必须部署云服务器。目前已有用户成功实践出可靠流程：**买云服务→装QMT→装miniQMT→装Python→装三方包→设置miniQMT交易日自动启动**。

但社区反映的真实痛点值得关注：

- **QMT客户端每次启动必须手动输入密码**，无法实现真正无人值守。有用户通过关闭Windows自动更新来维持长期在线。
- **QMT客户端崩溃会导致策略断连**。社区的建议是写一个守护进程（Watchdog），每分钟检查一次QMT客户端是否存活，挂了自动重启。

这些是在调研时容易忽略、但实盘中必须提前准备的问题。

### 3. QMT vs PTrade：你的技术栈更匹配QMT的方向

国内两大主流量化平台对比：

| 维度 | QMT/miniQMT | PTrade |
| :--- | :--- | :--- |
| **架构** | 本地代码运行，策略完全自主可控 | 代码上传至券商云端服务器执行 |
| **Python自由** | 可装任意第三方库 | 仅限平台预制白名单库 |
| **运行环境** | 支持Linux云端7×24小时挂机 | 依赖云端，本地关机策略照常运行 |

### 4. 数据源独立性：不能完全依赖miniQMT自带行情

miniQMT的xtdata模块只能获取基础行情数据，**不含Level-2逐笔成交、深度挂单等高级数据**。LOF溢价率实时计算真正依赖的是IOPV（实时预估净值）和场内实时价，这部分数据需要另外自行抓取。

建议：miniQMT只做交易执行通道，**行情监控层单独开发外挂**，用Tushare/AkShare等免费源做决策，独立于miniQMT运行。这样数据源可以被替代、被验证，不会因为某一方接口异常而错失机会。

### 5. 大数据量下的策略有效性：大额资金冲击

你目前的资金体量完全不会碰到流动性问题，单个LOF日成交额百万级别，你1万元进出是零摩擦的。**但如果以后本金扩大，单笔投入增加**，要注意：

- **必须加入流动性过滤**：日成交额必须大于单笔投入的10倍以上。
- **建议设置单次投入上限**：不超过日成交额的5%，避免卖出申购时自身冲击成本吃掉大部分利润。

### 6. 合规红线：个人开发者最容易踩的坑

2024年起监管对程序化交易实施了严格监控，以下组合容易触发异常交易预警，必须注意：

- ❌ **高频准入门槛**：单日申报、撤单笔数超20000笔，直接认定为高频交易，将面临重点监控。
- ❌ **虚假申报**：大量挂单后快速撤单，意图影响行情判断——被严格禁止。
- ✅ **你的LOF底仓套利不存在这些问题**：每天只有1-2笔卖出+申购动作，且非高频撤单策略，完全合规。
